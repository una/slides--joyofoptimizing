JPG

- PNG is an example of lossless compression, it doesn't lose quality. - PNG uses delta compression - difference of one pixel to another (bc most images have large areas of similar color) - chooses the pixel closest in value (hoping to be 0) which decreases overall file size - jpg also looks at difference of one pixel to another and discards the excess (it rounds where it can)
- jpg is really smart, it goes through a several-step process for compression - first it breaks up RGB into black and white (Y), red chroma (red-green), and blue chroma channels (blue-yellow) - goes through DCT to create 64 coefficients within this 8 pixel grid - then, goes through quantization -- which is based on a lookup-table corresponsing to image quality that you set when saving the jpg - convert to a new, compressed value, it's where the most visual information is lost - The upper left corner is the DC coefficient, which is just the unweighted average, and the others are called the AC coefficients, which really represent the difference in color detail - Runs through another algorithm to find averages and recompile color information and resaves the image - saves space on the server - lossy isn't a bad thing

Guetzli Compression

- and people are continuing to iterate on JPEG because its so widely used - Google *just* released Guetzli, a new open source algorithm last month - claims to produce high quality JPEG images, compressed 35% smaller than currently available method
- one on the right is guetzli compression, center is normal png compression - you can see very small differences in color, but its hard to see - Guetzli uses an AI-based compression metric - that incorporates the concept of cells in the retina to make decisions in quantization - the quantization table changes per image with Guetzli, it's not an absolute set of values like it normally is - tradeoff: images take longer to compress and results are unreliable - especially true on smaller images (works best as larger sizes)

.nginx-webp.conf

            
  user www-data;

  http {
    # Basic Settings
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;

    # image/webp webp;
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # Conditional variables
    map $http_accept $webp_suffix {
      default   "";
      "~*webp"  ".webp";
    }

    # Minimal server
    server {
      listen 80 default_server;
      listen [::]:80 default_server ipv6only=on;
      root /usr/share/nginx/html;
      index index.html;

      # Make site accessible from location
      # This ex is localhost
      server_name localhost;
      location ~* ^/images/.+\.(png|jpg)$ {
        root /home/www-data;
        add_header Vary Accept;
        try_files $uri$webp_suffix $uri =404;
      }
    }
  }
- We need to: 1. Determine a consistent naming scheme and location ( i.e. `unicorn.png` --> `unicorn.png.webp`) 2. Write script to read browser headers looking for an `HTTP Accept header` for WebP (or whatever format you choose) 3. Make sure to serve original files if there is no WebP counterpart

  • Be aware of emerging image formats (pros/cons)
  • <picture> is your friend!
  • Compress your smaller images to low qualities
  • Always run all images through an optimizer
  • Small improvements are still improvements
  • Educate the entire team
